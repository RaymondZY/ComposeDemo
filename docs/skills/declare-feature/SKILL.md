# 声明 biz 模块 feature.md

## 目标

编写 `biz` 模块下的 `feature.md`，只声明需求功能点，让实现和测试都能据此覆盖。不要在 `feature.md` 中写具体架构设计、类设计、方法名、字段名、DI 配置、测试函数名或代码组织方案。

## 文件位置

- 核心业务逻辑、状态规则、数据变换、业务约束、请求结果处理等平台无关需求，写入 `biz/<feature>/domain/feature.md`。
- 平台相关行为、用户交互、输入法/焦点、Compose UI 可见状态、Android 行为等需求，写入 `biz/<feature>/presentation/feature.md`。
- 如果现有 biz 子模块采用更细路径，例如 `biz/story/input/domain/feature.md`，沿用该模块当前路径。
- 不要把非 biz 模块纳入本 skill 的范围。

## 编写原则

- 只描述“要实现什么功能”和“用户或系统在什么条件下看到什么结果”。
- 每个 UC 是一个可实现、可验证的功能点；避免把多个独立行为塞进同一个 UC。
- 使用用户、系统、服务等业务语言；避免出现具体架构和代码细节。
- domain 的 feature 不写 UI 控件、颜色、图标、布局、焦点、键盘等平台细节。
- presentation 的 feature 不写复杂业务推导；需要业务判断时，描述为“根据业务状态/业务结果展示或交互”。
- 必须覆盖边界情况：请求失败、空数据、重复点击、重复请求、并发触发、超时、取消、恢复、无权限、离线等与该功能相关的场景。
- 占位能力要明确说明“可触发但暂无实际业务结果”，并写清楚不会破坏当前状态。

## 推荐结构

```markdown
# Feature — <模块名>（<中文说明>）

> 参考：<需求来源或模块职责，可选>

---

## 全局功能约束

- **<约束名>**：<对多个 UC 都生效的功能约束。>

---

## UC-01 <功能点标题>

**前置条件**：<触发前的状态或条件>  
**步骤**：

1. <用户或系统动作>
2. <系统响应>
3. <状态或结果变化>

**预期结果**：<可观察、可验证的结果>

---
```

## 编写流程

1. 确认目标是 biz 模块，并判断需求属于 domain、presentation，或需要拆到两层分别描述。
2. 阅读目标模块附近已有的 `feature.md`，保持标题、UC 编号和粒度一致。
3. 先列全局功能约束，只放跨 UC 生效的需求规则。
4. 按 UC 拆分主要功能路径，再补齐失败、重复触发、超时等边界 UC。
5. 审查每条内容，删除架构设计和代码实现细节。
6. 确认每个 UC 都能被实现和测试验证。

## 示例

```markdown
# Feature — Story Input（故事输入框）

> 参考：页面底部输入栏的展开、收起和输入行为。

---

## 全局功能约束

- **状态保留**：输入框收起或展开时，已输入文字不应被自动清空。
- **重复触发**：同一状态下重复触发展开或收起，不应造成状态错乱或重复副作用。

---

## UC-01 默认收起键盘，点击展开

**前置条件**：输入框处于收起状态（键盘未弹出）  
**步骤**：

1. 用户点击输入框区域
2. 系统请求软键盘弹出
3. 输入框进入展开状态，用户可自由输入文字

**预期结果**：键盘弹出，输入框获得焦点，可接收文字输入

---

## UC-02 已展开时重复点击输入框

**前置条件**：输入框处于展开状态（键盘已弹出）  
**步骤**：

1. 用户再次点击输入框区域
2. 系统保持当前输入状态

**预期结果**：输入框仍可继续输入，已有文字和光标位置不被异常重置

---

## UC-03 请求键盘弹出失败

**前置条件**：输入框处于收起状态  
**步骤**：

1. 用户点击输入框区域
2. 系统请求软键盘弹出
3. 键盘未能成功弹出

**预期结果**：输入框不进入错误的展开状态，用户可再次点击重试
```
